# BeatFlow — Security Strategy

## 1. Purpose

This document defines BeatFlow's security strategy.

Security must be considered from the beginning, not added at the end.

---

## 2. Authentication Security

Requirements:

- Hash passwords with bcrypt or Argon2id.
- Never store plain-text passwords.
- JWT access tokens must be short-lived.
- Refresh tokens must be revocable.
- Refresh token rotation should be supported.
- Login should return generic error messages.
- Auth endpoints should be rate-limited.

---

## 3. Authorization

Approach:

- RBAC
- ownership checks

Roles:

- guest
- buyer
- producer
- admin
- super_admin

Rules:

- Producer can update only own beats.
- Buyer can download only purchased files.
- Admin can moderate beats.
- Super admin can manage platform-level settings.

Future advanced options:

- Casbin
- OPA
- ABAC

---

## 4. API Security

Requirements:

- input validation
- CORS config
- secure headers
- request size limits
- consistent error responses
- no stack traces in production responses
- rate limiting

---

## 5. Webhook Security

Payment webhook requirements:

- verify signature
- store provider_event_id
- process idempotently
- reject invalid payloads
- do not trust frontend payment success redirect

---

## 6. S3 Security

Requirements:

- master files are private
- stems are private
- contracts are private
- downloads use short-lived presigned URLs
- object keys are generated by backend
- client cannot choose arbitrary object key
- IAM permissions follow least privilege

---

## 7. Container Security

Requirements:

- minimal base image
- multi-stage build
- no shell/tools in final image if not needed
- run as non-root user
- read-only filesystem when possible
- no secrets in image
- scan image with Trivy

Important principle:

- A production container should contain only what the application needs to run.

---

## 8. Kubernetes Security

Requirements:

- Secrets for sensitive config
- ConfigMaps for non-sensitive config
- least privilege service accounts
- resource limits
- avoid privileged containers
- avoid unnecessary exec access
- network policies later

Secret management future options:

- Sealed Secrets
- External Secrets Operator
- Secret Store CSI Driver

---

## 9. Static Security Analysis

Tools:

- gosec
- CodeQL
- Trivy
- Dependabot

CI should scan:

- source code
- dependencies
- Docker images

---

## 10. Audit Logging

Audit log required for:

- admin approve/reject/suspend
- payment status changes
- refund actions
- producer verification changes
- security-sensitive auth events

Audit fields:

- actor_id
- actor_role
- action_type
- target_type
- target_id
- reason
- metadata
- created_at

---

## 11. Sensitive Data Logging Rules

Do not log:

- passwords
- refresh tokens
- JWT secrets
- AWS secret keys
- payment provider secrets
- full private S3 URLs
- raw card data
- raw IP unless intentionally anonymized

---

## 12. Rate Limiting

Protect:

- login
- register
- refresh
- search
- payment webhook
- download URL generation
- message sending

Use Redis for distributed rate limiting.
